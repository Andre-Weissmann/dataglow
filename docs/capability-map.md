# DATAGLOW â€” Capability Map

A short, living index of what lives where in DATAGLOW's `js/` directory. Today
that directory is ~60 flat ES modules with no sub-folders, so "where does this
belong / where does that already live?" is a real question every coding session
hits. This file answers it without anyone having to load the whole codebase into
context.

**How to use it.** Read this index in full first â€” it is deliberately kept short
enough to do that cheaply. Each area lists what it owns and the exact file(s)
that back it. Once you've found the area your task touches, open only those
files. When an area needs more than a few lines to explain, the index links to a
companion detail file under [`capability-map/`](./capability-map/); load that
only when you're actually working in that area. That link-out-when-needed shape
is the whole point: the index stays skimmable, the depth stays out of the way
until you want it.

**Keep it honest.** When you add, remove, or repurpose a `js/` module, update the
relevant area below â€” and its entry in [`capability-map.manifest.json`](../capability-map.manifest.json) â€”
in the same PR (the same rule the [changelog](./CHANGELOG.md) and
[tech-debt tracker](./tech-debt-tracker.md) follow). A file that exists but isn't
mapped here â€” or a mapped file that no longer exists â€” is exactly the kind of
drift the CI **capability-map drift detector** (`npm run test:capdrift`,
`.github/scripts/capability-drift.mjs`) fails the build on, and that the weekly
[entropy-reduction scan](./entropy-reduction-scan.md) also surfaces.

> On provenance: keeping a short top-level index that points to deeper detail
> only when needed is a general, widely-used documentation practice. Everything
> here is DATAGLOW's own wording about DATAGLOW's own files.

## App shell & data engine
The core runtime every other module hangs off: tab wiring, shared state, DOM/format
helpers, the in-browser query engine, and file ingestion.
- **Controller & wiring** â€” `js/app-shell/main.js` (top-level app controller; wires tabs, events, and every feature module together).
- **State & helpers** â€” `js/app-shell/state.js` (central mutable app state), `js/app-shell/utils.js` (shared DOM, formatting, and hashing helpers).
- **Grouped tab navigation** â€” `js/app-shell/tab-groups.js` (flag: `groupedNavigation`, PROMOTED to ON): a pure reducer (`buildTabGroups`, `groupForTab`) that clusters the flat tab-id list into 5 named modes (Explore, Validate & Trust, Analyze, Visualize & Share, Automate) for `renderTabBar()` to render as grouped headers instead of one flat row. No DOM, no dependency on `main.js`; turning the flag off restores the original flat single-row bar, unchanged.
- **Validate tab focus mode** â€” `js/app-shell/validate-focus.js` (flag: `validateFocusMode`, PROMOTED to ON): pure disclosure-state logic (`shouldExpandAdvanced`, `createValidateFocusStore`) deciding whether the Validate tab's "Advanced options" block (context/domain-pack/level/export controls, AI Synthesis, Peer Review Mode) starts open or closed per dataset â€” closed until the analyst runs validation once or opens it manually. Turning the flag off renders the block always forced open, identical to pre-flag behavior.
- **Query engine** â€” `js/app-shell/duckdb-engine.js` (DuckDB-WASM engine; all SQL runs here, in-browser, zero upload).
- **File loading** â€” `js/app-shell/loaders.js` (CSV/TSV, JSON/NDJSON, Parquet, Excel, SQLite ingestion into DuckDB; also `loadRowsAsDataset` for in-memory result sets).
- **SQL editor highlighting** â€” `js/app-shell/sql-highlight.js` (dependency-free, zero-network SQL tokenizer/highlighter powering the SQL tab's overlay, plus structured DuckDB error formatting; pure, Node-testable functions).
- **Multi-dialect SQL translation** â€” `js/app-shell/sql-dialect-adapter.js` (Polyglot Workbench, Batch A; flag: `multiDialectSql`, PROMOTED to ON): a pure, dependency-free `translateDialectSql(sql, dialect)` that transpiles SQL written in PostgreSQL, MySQL, BigQuery, Snowflake, or T-SQL into the DuckDB SQL the engine runs, using a small set of composable regex/string rules that first mask string literals + comments so they can never be corrupted (same string-handling discipline as `sql-highlight.js`). `SUPPORTED_DIALECTS` drives the SQL-tab dialect-picker chip row. No DOM, no DuckDB import â€” Node-testable (`test/sql-dialect-adapter.test.mjs`, which also round-trips each dialect's translated output through a real DuckDB engine). When the flag is on, `main.js`'s `runSqlQuery()` runs `translateDialectSql(userSql, selectedDialect)` before `runQuery`; the default 'duckdb' selection is a no-op passthrough. When off no picker renders and the SQL tab is byte-for-byte unchanged. Batch B (an Object Space registry) is a separate follow-up.
- **Warehouse import** â€” `js/app-shell/databricks-connect.js` (proof-of-concept BYO-token, browser-direct read-only pull from a user's own Databricks SQL warehouse into DuckDB; see [`databricks-connect.md`](./databricks-connect.md)).
- **Capability registry** â€” `js/app-shell/capability-registry.js` (platform-aware module loader; reads each capability's `platforms` field from [`capability-map.manifest.json`](../capability-map.manifest.json), detects browser vs. Tauri desktop at runtime, and dynamically `import()`s only the modules meant for that runtime so `js/app-shell/main.js` no longer statically imports every feature).
- **Command Deck sidebar nav (Part 1)** â€” `js/app-shell/command-deck-nav.js` (pure content-model builder for an alternate left-sidebar nav that regroups the 13 real tabs into 5 Trust-Tier Lifecycle Stages â€” Frame/Work/Trust/Generate/Tell â€” instead of the flat top tab bar; `COMMAND_DECK_STAGES` is the static stage-to-tab mapping, `buildSidebarContent({tabMeta, activeTab})` resolves each mapped tab's real label/icon and marks the active one while honestly reporting any `unassignedTabs` rather than silently dropping them, `validateStageCoverage(realTabIds)` flags any missing or stale tab mappings, `stageForTab(tabId)` is a pure lookup). Wired into `js/app-shell/main.js` as `renderCommandDeckSidebar()`, called from `init()` and at the end of `switchTab()` so the alternate sidebar's active-tab highlight and collapsed/expanded stage state stay in sync with every tab switch, rendering into the new `#command-deck-sidebar` host in `index.html` (sits alongside the existing `<nav class="tabbar">`, not inside `#data-sidebar` which is unrelated dataset-loading UI). Behind the `dataglowSidebarNav` flag, PROMOTED to ON â€” `#command-deck-sidebar` now renders by default alongside the existing top tab bar for every user; turning the flag off restores the prior behavior exactly (`renderCommandDeckSidebar()` hides the host and renders nothing, and the top tab bar remains the app's one and only nav, byte-for-byte unchanged). Direction, scope (Part 1 only â€” command palette and adaptive next-step rail are separate future parts), and naming were decided against the UI/UX brainstorm report per the user's "build all, safely and smartly" instruction; the full rationale is recorded as code comments at the top of `command-deck-nav.js`. A dedicated test (`test/command-deck-nav.test.mjs`) regex-extracts the real `TAB_META` block straight out of `main.js` at test time, so the stage mapping can never silently drift out of sync with the app's actual tool list.
- **Shared metrics registry** â€” `js/app-shell/metrics-registry.js` (an in-session "define once" source of truth for named metric definitions â€” e.g. `revenue` â†’ `SUM(amount)` â€” so every consuming surface references the same business term the same way within one session. Pure, dependency-free: `createMetricsRegistry()` returns `defineMetric`/`getMetric`/`hasMetric`/`listMetrics`/`removeMetric`/`resolveMetricSql`/`fingerprint`, and the standalone `expandMetricReferences(sql, registry)` rewrites `@name` query tokens into each metric's compiled, parenthesized SQL fragment. It only NAMES read-only SQL expressions evaluated over the active dataset's DuckDB table â€” it never runs SQL or mutates data, and DuckDB stays the sole compute engine. A fresh registry is keyed per dataset table name in `js/app-shell/state.js` (`getMetricsRegistry`/`getActiveMetricsRegistry`), mirroring the per-table provenance chains; the fingerprint reuses `sha256Hex` from `js/provenance/provenance.js` via a lazy import. The SQL tab is the first wired surface â€” a "Saved Metrics" card in `js/app-shell/main.js` defines/lists metrics and inserts their compiled SQL on an explicit click, and `runSqlQuery` expands `@metric` references before executing. Python/R/Visualize/Story tabs are deferred follow-ups (see [`tech-debt-tracker.md`](./tech-debt-tracker.md)).
- **Command Deck command palette (Part 2)** â€” `js/app-shell/command-palette.js` (pure logic, no DOM): `COMMAND_ACTIONS` is a small static registry of common in-tool actions (run query, run validation, scan for cleaning issues, run preflight, run diagnostics, export as an Excel workbook), each naming a stable `run` id the caller dispatches to a real function -- this module never calls anything itself. `buildCommandList({tabMeta, tabOrder, activeTab})` emits one "tab" command per real tab (resolved from the caller's real `TAB_META`/`tabOrder`, so it can never silently drift out of sync with the app's actual tool list) plus every action command whose optional `whenTab` matches the current tab (actions with no `whenTab` are always offered). `scoreCommand`/`filterCommands` rank matches best-first: exact match > prefix > substring > keyword > fuzzy subsequence, empty query matches everything weakly so the palette can show a default list. Wired into `js/app-shell/main.js` as a global Ctrl/Cmd+K listener (`initCommandPalette()`, called from `init()`) that opens a modal (`#command-palette-overlay` in `index.html`, styled in `css/app.css`), live-filters as the user types, and supports arrow-key navigation plus Enter-to-choose and Escape-to-close; choosing a "tab" command calls `switchTab()`, choosing an "action" command calls the matching real function via `runPaletteAction()`. Ships dark behind the `dataglowCommandPalette` flag (default OFF) -- with the flag off, `openCommandPalette()` no-ops and no listener changes behavior. Independent of Part 1's sidebar (either flag can be on/off without affecting the other). A dedicated test (`test/command-palette.test.mjs`) regex-extracts the real `TAB_META`/`tabOrder` straight out of `main.js`/`state.js` at test time, same drift-proofing discipline as Part 1.

> **Platforms.** Every capability in [`capability-map.manifest.json`](../capability-map.manifest.json)
> declares a `platforms` list drawn from `browser`, `desktop`, and (reserved) `mobile`.
> Most capabilities work identically in a plain browser and inside the Tauri desktop
> shell and so are `["browser", "desktop"]`; the Watch Folder is browser-only (its File
> System Access polling has no counterpart in the desktop shell) via a per-file
> `platformsByFile` override. The capability-map drift gate (`npm run test:capdrift`)
> fails the build if any capability is missing a valid, non-empty `platforms` list.

## Universal ingestion & RAG (wave 2)

- **Universal Drop Zone router** â€” `js/drop-zone/drop-zone-router.js`: the pure-logic routing layer between a file-drop event and DataGlow's ingestion pipeline, with NO browser APIs (no `File`, no `FileReader`, no drag/drop handling â€” the UI layer reads bytes first and passes plain data in). `detectFileFormat(fileName, mimeType, firstBytes)` detects a file's format using magic bytes first (e.g. `PAR1` for Parquet, `%PDF` for PDF), then MIME type, then filename extension as a last resort, returning `{format, confidence, handler}`. `buildDropManifest(files)` turns an array of dropped files into a `DropManifest` â€” one item per file with detected format/handler/confidence, a cleaned display name, and tab order â€” plus manifest-level flags (`hasMixedFormats`, `requiresTranscription`, `requiresRAG`). `routeDropManifest(manifest)` partitions the manifest's items into per-handler buckets (`duckdbFiles`, `univerFiles`, `ragFiles`, `transcriptionFiles`, `webCodecsFiles`, `unknownFiles`) so the app shell can dispatch each file to the correct subsystem. `buildJoinPanelDescriptor(loadedDatasets)` is a heuristic-only join-suggestion aid for a multi-file drop â€” it proposes candidate join pairs between loaded datasets by column-name overlap but never executes a join automatically; the user always reviews and confirms a suggestion first. `buildTabDescriptor(fileId, displayName, format, status)` builds the per-file tab-strip descriptor shown while ingestion is in progress. Format coverage now also includes `.txt`/`.log` (routed to DuckDB via the line parser below), Arrow IPC/Feather v2 (detected by the `ARROW1` magic-byte header, routed to DuckDB), and `.xml` (routed to RAG/document mode rather than falling into `unknown`). `jsonNeedsFlattening(format)` tells the app shell whether a `.json`/`.ndjson` file should be pre-processed through `js/ingestion/json-flattener.js` before DuckDB ingestion.
- **RAG core (chunker, cosine similarity, retrieval)** â€” `js/rag/rag-core.js`: pure JavaScript logic with zero network/DOM dependency, runnable identically under plain Node and inside the browser. `chunkText(text, chunkSize=200, overlap=20, source='unknown')` splits raw text into overlapping, whitespace-word-count-approximate chunks (a cheap approximation for LLM tokens, not a real tokenizer). `cosineSimilarity(vecA, vecB)` computes cosine similarity between two equal-length numeric vectors, returning 0 rather than dividing by zero when either vector has zero magnitude. `retrieveTopK(queryVec, knowledgeBase, k=2)` ranks an in-memory knowledge base of `{id, text, source, embedding}` entries against an already-computed query vector and returns the top-k matches sorted by score. This module never computes an embedding itself â€” that is deliberately left to the transformers.js WebWorker bridge elsewhere in the app, injected by the caller.
- **RAG validation bridge (citation injection)** â€” `js/rag/rag-validation-bridge.js`: wires validation findings to the RAG knowledge base so findings can be annotated with a supporting domain-knowledge citation. `attachCitationsToFindings(findings, knowledgeBase, queryVecFn, minScore=0.65)` calls the caller-injected `queryVecFn(finding.message)` to embed each finding's message, uses `retrieveTopK` from `rag-core.js` to find the single best-matching knowledge base entry, and attaches a `citation: {text, source, score}` to the finding only when the match's cosine similarity exceeds `minScore` â€” otherwise `citation` is set to `null`. Findings are mutated in place and returned for convenience. The embedding function is dependency-injected so the module's tests never need a real embedding model, a WebWorker, or a network call.
- **User Knowledge Store (in-memory RAG index)** â€” `js/rag/user-knowledge-store.js`: lets an analyst paste or upload their own domain reference documents (SOPs, coding manuals, policy docs) and have them chunked and indexed locally for RAG retrieval. Pure in-memory logic â€” no `fetch`, no `localStorage`, no other browser-only API; state lives in a single in-memory array for the page's lifetime. `addUserDocument` chunks a document via `rag-core.js`'s `chunkText` and appends it to the knowledge base with a `null` embedding (set later by the caller); `getUserKnowledgeBase` returns the current in-memory index; `clearUserKnowledge` empties it; `getUserKnowledgeStats` summarizes chunk/source counts; `setChunkEmbedding(id, vector)` is called by the WebWorker embedding bridge once a chunk's vector is computed, returning `false` for an unknown id rather than throwing.
- **Audio ingestion structurer (Whisper â†’ structured transcript dataset)** â€” `js/audio/audio-structurer.js`: pure JavaScript, browser-free, network-free â€” step 6 of the audio ingestion pipeline (file drop â†’ ArrayBuffer decode â†’ Web Worker â†’ transformers.js Whisper transcription â†’ **this module** â†’ DuckDB load). `validateTranscriptionInput(whisperSegments)` checks the raw `{timestamp:[startSec,endSec], text}` segment array Whisper's pipeline produces is well-formed before structuring. `structureTranscription(whisperSegments, sourceFileName, options)` turns those segments into the same `{columns, rows}` grid shape every other ingest path (CSV, Excel, SQLite, â€¦) hands to the DuckDB loader â€” one row per segment with `segment_id`/`start_sec`/`end_sec`/`duration_sec`/`text`/`char_count`/`word_count`/`words_per_minute`, plus a scaffolded `speaker` column (`options.speakerColumn`, defaulting to a `'SPEAKER_00'` placeholder) so a future diarization model can be dropped in without changing the dataset shape. `buildAudioDatasetSummary(structuredDataset)` builds the display-ready tab-strip summary (segment count, total duration, word count). See `js/audio/whisper-worker.scaffold.js` for the browser-only Worker pattern that produces the raw segments this module consumes (transformers.js Whisper, WebGPU with WASM fallback, 16kHz mono resampling) â€” a scaffold file to be copied into the real build's Worker rather than imported directly, since it documents `importScripts`/`postMessage` wiring that has no meaning in a plain Node import graph.
- **Video ingestion bridge (audio-only, Batch 1)** â€” `js/video/video-ingestion-bridge.js`: pure-logic helpers only â€” no DOM, no WebCodecs calls, no file I/O â€” that re-use the existing audio pipeline for video files' spoken-word content rather than building a second transcription engine or dataset shape. `buildVideoManifest(fileName, fileSizeMb, durationHint)` documents the shape of the video ingestion manifest ahead of extraction. `validateVideoFile(fileName, mimeType, fileSizeMb)` validates a candidate file before it reaches the browser-only audio-extraction step. `estimateTranscriptionTime(durationSec, hasWebGPU)` estimates how long Whisper transcription will take given WebGPU availability. `buildVideoTranscriptDatasetName(fileName)` derives the resulting transcript dataset's name. Batch 1 is audio-only by design: `extractionMode` is always `'audio_only'`, and `frameExtractionStatus` honestly reports `'not_implemented'` rather than implying a vision-captioning capability that doesn't exist yet. See `js/video/webcodecs-audio-extractor.scaffold.js` for the browser-side pattern (`AudioContext.decodeAudioData` / WebCodecs `AudioDecoder`) that extracts the Float32Array audio this module's pipeline hands to the existing Whisper Worker (`js/audio/whisper-worker.scaffold.js`) â€” a scaffold documenting two extraction patterns (simple whole-file decode vs. WebCodecs streaming) for the UI layer to adapt, not imported directly.
- **PDF ingestion bridge (PDF.js â†’ RAG pipeline)** â€” `js/pdf/pdf-ingestion-bridge.js`: wires PDF files dropped onto the Universal Drop Zone into DataGlow's RAG pipeline (`js/rag/rag-core.js` + `js/rag/user-knowledge-store.js`). Pure JavaScript logic only â€” it does not import PDF.js and does not touch the DOM, `fetch`, or any browser-only API; PDF text extraction happens upstream in the UI layer (see the scaffold below), and by the time this module runs the PDF's text is already a plain JavaScript string. `buildPDFManifest` builds a display manifest describing the incoming PDF. `validatePDFText` checks the extracted text is actually usable before indexing (a fully-scanned, image-only PDF with no extractable text is flagged rather than silently indexed empty). `preparePDFForRAG` chunks the text via `rag-core.js`'s `chunkText` and adds it to the in-memory RAG knowledge base via `user-knowledge-store.js`'s `addUserDocument`. `buildPDFIngestionSummary` builds the display-ready summary for the tab strip / chat UI (chunk count, word count). See `js/pdf/pdfjs-extractor.scaffold.js` for the browser-side text-extraction pattern this module's input comes from â€” PDF.js loaded from a CDN (Apache 2.0, Mozilla), extracting all text from text-based PDFs client-side with no server/upload; image-only scanned PDFs are a documented limitation (OCR not yet supported). A scaffold file, not imported directly â€” it documents the pattern for the UI layer's actual PDF.js wiring.
- **Text / Log line parser (.txt and .log to queryable rows)** â€” `js/ingestion/text-line-parser.js`: pure JavaScript, no browser APIs, no async â€” converts plain text (`.txt`, `.log`) into a queryable row dataset the same way every other ingest path hands rows to the DuckDB loader. `parseTextLines(text, opts)` splits the raw string into one row per line (`{line_number, content}`), preserving blank lines by default so line numbers stay accurate, with an optional `skipEmpty` flag and a `maxLines` safety cap (default 500,000). `inferTextKind(sampleLines)` looks at the first 20 lines and classifies the file as `'log'` (timestamp or severity-level patterns), `'delimited'` (tab/comma heavy â€” probably mis-named CSV/TSV), `'prose'` (long freeform lines), or `'unknown'`. `buildTextDataset(parsed, fileName, kind)` wraps the parsed rows into the `{columns, rows, meta}` shape the Canvas shell expects, with a `meta.note` contextual hint tailored to the detected kind (e.g. suggesting `LIKE`/`REGEXP` queries on the `content` column for logs). `.txt` and `.log` route through DuckDB via `js/drop-zone/drop-zone-router.js`.
- **Semi-structured JSON flattener (nested JSON, FHIR bundles, API envelopes)** â€” `js/ingestion/json-flattener.js`: pure JavaScript, no browser APIs, no side effects â€” normalises nested JSON into the flat array of row objects DuckDB can ingest. `flattenJson(parsed, opts)` handles four shapes: an already-flat array of objects (passthrough, `confidence: 'high'`), a single object wrapping an array (GitHub `/repos`-style or Stripe `charges.data`-style envelopes, found via BFS and reported with `confidence: 'medium'` plus a `warning` noting which key was extracted and that sibling keys were discarded), a deeply nested single object with no array inside (treated as one row, `confidence: 'low'`), and an array of scalar values (each becomes an `{index, value}` row). Nested objects are dot-key expanded up to a configurable `maxDepth` (default 4); arrays found in a leaf position are JSON-stringified rather than silently dropped. `parseJsonOrNdjson(text)` tries standard `JSON.parse` first, then falls back to parsing the text as newline-delimited JSON (one object per line), returning `{parsed, isNdjson, error}`. `jsonNeedsFlattening(format)` is the small routing helper `js/drop-zone/drop-zone-router.js` calls to decide whether a `.json`/`.ndjson` file should pass through this flattener before DuckDB ingestion (currently always true â€” the flattener is a no-op passthrough for already-flat data).
- **Image OCR (Tesseract.js, client-side, zero upload)** â€” `js/ingestion/image-ocr.js`: pure JavaScript, no browser APIs in its tested exports â€” extracts text from `.png`/`.jpg`/`.jpeg`/`.webp`/`.bmp`/`.gif` files dropped onto the Universal Drop Zone via Tesseract.js running entirely in the browser (Web Worker, zero upload, zero cloud call, no API key). `isImageFile(fileName)` checks the extension against the supported image list. `parseOcrText(rawText, opts)` turns Tesseract's raw text output into the same `{line_number, content}` row shape `text-line-parser.js` produces, with an optional `skipEmpty` flag. `inferOcrKind(lines)` classifies the OCR'd content as `'table'`, `'form'`, `'code'`, `'prose'`, `'mixed'`, or `'unknown'` by pattern-matching column spacing, `Label:` pairs, and code keywords. `scoreOcrConfidence(confidences)` grades Tesseract's word-level confidence array into `{mean, low, grade}` (`'high'`/`'medium'`/`'low'`/`'poor'`). `buildOcrDataset(parsed, fileName, kind, confidenceScore)` wraps the parsed rows into the `{columns, rows, meta}` shape the Canvas shell expects, with a `meta.note` warning when confidence is low. `js/drop-zone/drop-zone-router.js` detects images by magic bytes (PNG `89 50 4E 47`, JPEG `FF D8 FF`, BMP `42 4D`, GIF `47 49 46`) or extension fallback and routes them to the new `'ocr'` handler; `isImageFormat(format)` is the small routing helper. The browser-only Tesseract.js v5 CDN loader and `runOcr(imageFile)` call live inline in `canvas/index.html` (not exported from this module, not unit-tested â€” same convention as the Whisper/PDF.js CDN loaders) and drive a `'transcribing'` tab status while OCR runs, followed by a confidence-grade badge once ready.
- **Live API / Webhook feed (REST endpoint fetch with polling, auto-normalization)** â€” `js/ingestion/api-feed.js`: pure logic, no `fetch()`, no DOM â€” the Canvas agent bar's "âšˇ Live Feed" panel calls this module to validate input and normalize a response the browser layer already fetched; the actual `fetch()` call is made from the browser directly against the user's endpoint, so DataGlow's servers never see or proxy the response (zero-upload guarantee, same posture as every other client-side ingestion path). `validateFeedUrl(url)` checks a candidate endpoint URL, auto-prepending `https://` when the scheme is omitted, and rejects anything other than `http:`/`https:` (e.g. `ftp://`), returning `{valid, error, normalized}`. `parseHeadersString(raw)` turns a textarea's worth of `Key: Value` lines into a plain headers object for the `fetch()` call, silently skipping malformed lines. `normalizeApiResponse(parsed, opts)` mirrors `json-flattener.js`'s contract for the three shapes an API response is likely to take â€” flat array of objects (`confidence: 'high'`), a single object wrapping an array (API envelope, BFS-found and reported with the extraction `path`), or a single object with no array (treated as one row) â€” with the same dot-key flattening and `maxRows` cap. `buildPollSchedule(intervalMs, method, url, headers)` builds the descriptor the UI layer uses to drive `setInterval`-based re-fetching (`0` = one-shot; 5s/15s/30s/1min/5min presets, with a generated label for any other interval), defaulting an invalid method to `GET`. `buildFeedDataset(rows, url, pollSchedule, fetchedAt)` wraps normalized rows into the same `{columns, rows, meta}` shape every other ingestion path (`text-line-parser.js`, `json-flattener.js`) hands to the DuckDB loader, with `meta.format = 'api'` and a `meta.note` describing the source URL and refresh cadence. In the Canvas UI, a fetched feed becomes a new tab with an orange "API" format badge, the URL's hostname as the tab name, and â€” when polling is active â€” a pulsing green live indicator dot; new data flows through the exact same validation pipeline (`runRealValidation`) as a dropped file.


## Question Prompter (Feature 13 â€” "Where to start" intelligence)
The question-first analysis layer: DataGlow generates 3â€“7 contextually relevant business questions from a dataset's file name, inferred schema, validation findings, and column statistics â€” before the analyst has to know what to ask.
- **Question Prompter** â€” `js/questions/question-prompter.js`: pure-logic, rule-based (not LLM-dependent), Node-testable module with no browser APIs and no network calls. `generateQuestions(findings, columnStats, options)` (post-validation mode) turns validation findings and column statistics into ranked `Question` objects, firing template rules for high nulls (>10%), right skew (skewness >2), outliers, low cardinality, date gaps, join fanout, schema drift, and domain-specific patterns (e.g. healthcare length-of-stay), respecting `options.domain`/`maxQuestions`/`includeSQL`. `generatePreUploadQuestions(fileName, columnNames, options)` (pre-upload mode) generates 3â€“5 questions from a file name and column names alone â€” before any data loads â€” by pattern-matching date-like/amount-like/id-like column names and inferring the likely table grain. `inferDomain(fileName, columnNames)` scores column/file-name tokens against healthcare/finance/retail/hr/operations signal lists and returns the best-matching domain (or `'general'`); callers can override inference by passing an explicit `domain` option to any other function. `updateStreamingQuestions(existingQuestions, newBatchFindings, batchNumber)` (streaming mode) diffs each new NATS batch's findings against the current question set, returning `{questions, newQuestions, resolvedQuestions}` so the Canvas rail can surface newly-revealed questions and retire ones whose triggering condition has cleared. `rankQuestions(questions, previouslyShown)` sorts high-priority-first and never repeats a question already shown. `generateStarterSQL(question, tableName, columnStats)` returns a runnable SQL query tailored to the question's originating template (e.g. a `COUNT(*) - COUNT(col)` null check, a `GROUP BY ... HAVING cnt > 1` duplicate-key scan for fanout). `formatQuestionsCard(questions)` renders the plain-text "WHERE TO START" block the Canvas card displays, with priority tags and starter SQL/action per question. Question ids are deterministic (a stable hash of template + triggering context), so the same findings always produce the same question set. See [`docs/question-prompter.md`](question-prompter.md) for the full template library, integration guide, and philosophy.
