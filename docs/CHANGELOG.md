- Add an Export / Reporting layer that turns the loaded, validated dataset into a downloadable Excel workbook or PDF report, 100% client-side. A Universal Export Contract (`js/export-report.js`) builds raw bytes per format decoupled from delivery: the `.xlsx` path reuses the already-vendored SheetJS (`assets/xlsx/`, no new dependency) and emits a Data sheet plus a Summary sheet and â€” when validation metadata is present â€” a Validation Summary sheet; the PDF path is a small first-party, dependency-free PDF 1.4 writer (catalog/pages/font objects + byte-offset xref table, ASCII-safe text) carrying the title, generation timestamp, row/column counts and validation summary. Delivery is handled by small platform adapters (`js/export-delivery.js`): the browser adapter uses the repo's existing Blob + object-URL + synthetic `<a download>` pattern; the desktop adapter feature-detects the Tauri `dialog.save`/`fs.writeBinaryFile` APIs for a native Save-As and transparently falls back to the browser download when they're absent (the shell's current deny-by-default posture), and reports a clean not-delivered/cancelled result when the user cancels the dialog; the mobile adapter is an explicit future-work share-sheet stub that throws. The module is registry-native (`export-reporting` capability in `capability-map.manifest.json`, `platforms: ["browser","desktop"]`) and reached from `js/main.js` via `registry.get('export-report')`; UI entry points are two buttons in the Visualize tab's new Export / Report card. A source guard in the unit tests (`npm run test:export`, new `export-reporting` CI job) asserts neither module contains a network primitive, keeping the zero-upload promise intact. Why: users need to get their cleaned data and its validation verdict out of the browser as a shareable artifact without any upload path.
- Add a platform-aware capability registry so `js/main.js` no longer statically imports every feature module regardless of runtime. Each capability in `capability-map.manifest.json` now declares a `platforms` list drawn from a closed set (`browser`, `desktop`, reserved `mobile`); most are `["browser", "desktop"]` (they behave identically in a plain browser and inside the Tauri desktop shell), and the Watch Folder is browser-only via a per-file `platformsByFile` override (its File System Access polling has no desktop-shell counterpart). A new loader `js/capability-registry.js` reads that manifest at runtime (same-origin `fetch`, precached by `sw.js` and staged into the desktop bundle â€” no new network/upload path), detects browser vs. Tauri (`window.__TAURI__`), and dynamically `import()`s only the modules whose platform list includes the detected runtime, exposing `registry.get(name)`/`has`/`available`/`list`. `js/main.js` migrates 13 representative feature modules onto the registry (a browser-only one, universals) while the rest keep working via their existing static imports; the remaining modules are tracked in a follow-up issue. Requesting a wrong-platform or unknown capability returns `undefined` with a clear `console.warn` (the runtime safety rail) rather than crashing â€” see the Watch Folder guard in `initWatchFolder`. The capability-map drift gate (`npm run test:capdrift`) now fails the build if any capability is missing a valid, non-empty `platforms` list, and `npm run test:capregistry` unit-tests the loader (both in the `capability-map-drift` CI job). Why: give each runtime only the modules that make sense for it, catch platform misclassification at merge time, and stop main.js from eagerly loading code a given runtime can't use.
