- Add the Governed Synthetic Data Passport (Trust Passport, Batch 4 — completing the four-batch Trust Passport concept): when a dataset leaves DATAGLOW as SYNTHETIC (via the Synthetic Twin, `js/privacy/synthetic-twin.js`, or the DP aggregate export path, `js/privacy/privacy-budget.js`) it can carry a full governance record instead of naked numbers. New pure, browser-free, network-free module `js/privacy/synthetic-data-passport.js` COMPOSES the three prior batches and adds nothing to their crypto or shapes: it calls Batch 2 (`buildDataNutritionLabel`) with `isSynthetic:true` as the container, carries the Batch 1 source-data Semantic/Metrics Layer checks/custody/assumptions through that label unchanged, and offers optional Batch 3 (`sealCheckResult`/`attachSealToLabel`) tamper-evidence. `buildSyntheticDataPassport(ctx)` returns `{kind:'dataglow-synthetic-data-passport', schemaVersion:1, generatedAt, label, synthetic, disclaimer}` where the `synthetic` block honestly describes HOW the data was generated and WHAT privacy guarantee (if any) applies; it REFUSES to build with no `ctx.generation` record. HONEST NAMING is the hard constraint: `describeSyntheticGeneration` asserts a formal differential-privacy guarantee (mechanism + a specific ε) ONLY when the generation context genuinely establishes one (a recognized `dataglow-synthetic-twin` kind, or a Laplace/DP mechanism string, each with a positive ε, or an explicit caller `formalDifferentialPrivacy:true` with a positive ε); a bare ε, a heuristic generator, or an explicit `formalDifferentialPrivacy:false` all yield "no formal guarantee — treat as potentially re-identifiable"; it NEVER upgrades a DP budget to "anonymized", a HIPAA Safe Harbor, an Expert Determination, or any legal/clinical determination, and it carries the generator's own disclaimer verbatim. `sealSyntheticPassport(passport, context)` is ALWAYS an explicit opt-in caller action (nothing seals on its own): it seals the exact generation parameters bound to a SHA-256 fingerprint of the synthetic OUTPUT, anchors the seal to the label's `custodyChain.finalHash`, attaches it additively (new `custodyChain.seals` array, no Batch-2 field changed), and returns a NEW passport (input not mutated); it refuses to mint a seal with no data fingerprint. Also exports `renderPassportSummaryLines`/`renderPassportSummary` (leads with the honest privacy line so it can never be separated from the numbers) and `exportPassportAsJSON` (round-trips losslessly). Wired opt-in into the Synthetic Twin export UI behind a new OFF-by-default `syntheticDataPassport` flag in `flags.manifest.json`; Batches 1–3 modules are untouched (additive only). See `test/synthetic-data-passport.test.mjs`.
- Add the Verifiable Check Seal ("Proof-of-Clean", Trust Passport, Batch 3): seal a validation check result into a portable, offline-re-verifiable artifact that proves the check ran against data matching a fingerprint and produced a given result — WITHOUT revealing the data. New pure, browser-free, network-free module `js/provenance/verifiable-check-seal.js` does NOT invent crypto: it APPLIES the Merkle-tree (SHA-256) commitment already in `js/provenance/selective-disclosure-proof.js` (reusing `hashLeaf`/`buildMerkleTree`/`merkleProof`/`rootFromProof` over the shared `sha256Hex` primitive). `sealCheckResult(result, context)` commits a fixed set of claims — check name/kind, a params fingerprint (e.g. the SQL text), a SHA-256 `data` fingerprint (the ONLY thing about the raw data that enters the artifact), dataset identity + columns fingerprint, and the result status/flag-count/full-result fingerprint — and returns `{kind:'dataglow-verifiable-check-seal', version:1, generatedAt, algorithm, fingerprintAlgorithm, check:{name,kind}, result:{status,flagCount}, dataglow, labelAnchor, fingerprints:{data,dataSource,params,result,columns}, commitment:{merkleRoot,leafCount,leafHash,nodeHash}, disclosedClaims:[{index,type,subject,value,statement,proof}], disclaimer}`. It REFUSES to mint a seal with no data binding (no empty-guarantee seals). `verifySeal(seal, data?)` genuinely re-checks two independent layers: (1) it re-derives every disclosed claim's leaf hash and folds it up its Merkle path to the committed root, so any altered sealed value fails; and (2) when data is supplied it re-fingerprints it and compares to the committed `data` fingerprint, so data modified since sealing FAILS to match (`dataMatch:false`) — a real tamper check, reported as `null` (not silently "passed") when no data is given. `attachSealToLabel(label, seal)` is ADDITIVE and non-mutating: it returns a new batch-2 Data Nutrition Label with the seal appended to a new `custodyChain.seals` array (anchored to the label's `custodyChain.finalHash`) and every existing custodyChain field (`algorithm`/`length`/`finalHash`/`steps`) preserved byte-for-byte. `renderSealSummaryLines(seal)`/`exportSealAsJSON(seal)` give a human-readable summary and a stable JSON artifact. HONEST NAMING (the batch's most important constraint) is enforced in code, in `CHECK_SEAL_DISCLAIMER`, and by a source-guard test: it is explicitly NOT a zero-knowledge proof (parameters/result/fingerprints are cleartext; only the raw data stays private), NOT a certification, NOT "blockchain", and never "certified" — it does not attest the data is accurate or truthful, only that the check ran against the fingerprinted data and produced this result. Wired into the SQL tab's Local Analysis Contract flow as strictly OPT-IN: with `verifiableCheckSeal` on (and `localAnalysisContract` on), a "Seal this result" button appears — a human must click it for anything to be sealed; it never seals automatically, and offers the `.json` seal as a client-side Blob download. Ships behind the new `verifiableCheckSeal` flag in `flags.manifest.json` (OFF by default); with the flag off, no seal UI renders and behaviour is exactly as before this PR. Tests: `npm run test:checkseal` (`test/verifiable-check-seal.test.mjs`, new `verifiable-check-seal` CI job, 54 assertions) seals a REAL Local Analysis Contract result, verifies the commitment from the artifact alone, confirms a matching-data check passes, asserts a TAMPERED-data check FAILS and an artifact-tampered seal fails the commitment, exercises the precomputed-fingerprint path, attaches a seal to a real batch-2 label and proves the existing fields are untouched, and includes zero-upload + honest-naming source guards. Registered as the `provenance-check-seal` capability. Known limitation (tech-debt tracker): the SQL-tab affordance fingerprints the query RESULT rows (the data in hand), so the seal binds to that query's output rather than the whole source table, and full-result fingerprinting is unbounded in size — both stated plainly in the seal's disclaimer and the tracker. Why: give an ordinary analyst a way to prove locally, with no cloud service and no crypto wallet, that a specific data-quality check ran on the real data and produced a specific result — the verifiable-computation piece Batch 4's synthetic-data passport builds on — without any upload path and without over-claiming what the mechanism is.
- Add the Data Nutrition Label (Trust Passport, Batch 2): a portable, opt-in provenance manifest for exports. New pure, browser-free, network-free module `js/provenance/data-nutrition-label.js` ASSEMBLES a self-describing JSON manifest from the data already tracked elsewhere — it does not duplicate any of it. `buildDataNutritionLabel(ctx)` reads the EXISTING tamper-evident chain of custody (accepting a trail array, a chain object with `getTrail()`, or `{steps}`), the Assumption Ledger, and validation check results, and returns `{kind:'dataglow-data-nutrition-label', schemaVersion:1, generatedAt, dataset:{name,table,rowCount,columnCount,columnNames,loadedAt}, checksRun:[{id,name,status,summary}], findingsSummary:{total,bySeverity:{pass,warn,fail,idle}}, transformations:[{step,op,description,at}], assumptions:[{at,source,action}], isSynthetic:false, custodyChain:{algorithm,length,finalHash,steps:[…]}, disclaimer}`. `custodyChain` is surfaced as an OBJECT so `finalHash` sits at the top level for Batch 3 (selective-disclosure proofs) to anchor against; `transformations` is a lightweight, explicitly-derived human projection of the custody steps (the hashes in `custodyChain` remain authoritative). `isSynthetic` is present and defaults false for Batch 4 (synthetic-data metadata) to set true. Wired into the export flow as strictly OPT-IN: a person must tick a new checkbox (`#export-include-label`, revealed only when the flag is on) — the label is never silently attached. `js/app-shell/main.js` builds the label on export, renders its lines into the workbook/report (a new "Data Nutrition Label" sheet / an appended text block via `js/export/export-report.js`, which stays decoupled — main.js passes rendered lines in, the export module imports no provenance code), and separately offers the `.json` manifest as its own client-side Blob download. Tests: `npm run test:nutritionlabel` (`test/data-nutrition-label.test.mjs`, new `data-nutrition-label` CI job) builds a manifest from a REAL provenance chain, checks every field and hash, exercises the sparse-context and trail-array paths, round-trips the JSON, asserts the flag-off export passthrough is identical while flag-on adds the sheet, and includes a zero-upload source guard (no `fetch`/`XMLHttpRequest`/`WebSocket`/`EventSource`/`sendBeacon`). Registered as the `provenance-nutrition-label` capability. Schema versioning is provisional (integer `schemaVersion:1`, matching the sibling provenance modules' `version:1` convention rather than the `protocol/` semver) — noted in the tech-debt tracker along with the JSON being delivered as a separate download rather than embedded in the page. Why: give anyone a portable, honest, offline record of where an exported dataset came from, what was checked, and what was assumed — the shared manifest shape Batches 3 and 4 build on — without any upload path and without ever auto-attaching itself.

- Add the Semantic Drift Watchdog, a de-duplication/notification-worthiness layer over the distribution-drift validation the engine already computes on every re-check, so the Watch Folder's automatic poll loop stops re-showing the SAME drift finding on every unchanged re-validation. New pure, dependency-free, Node-testable module `js/ambient/drift-watchdog.js` exports `summarizeDriftEvent(drift)` (normalizes the existing `distribution_drift` layer result — including its Holt-forecast alert flags — into a `{severity, headline, lines}` shape, degrading to a silent pass on missing/malformed input, never throwing), `alertFingerprint(summary)` (a stable, content-based, line-order-independent hash), the `DriftWatchdog` class (`.observe(fileName, drift)` → `{summary, isNew, shouldNotify}`, tracked per file name so an unrelated file's drift never suppresses another's, plus `.clear`/`.clearAll` to re-arm), and `formatWatchdogAlert(fileName, decision)` for a one-line log/toast surface. It computes zero new statistics of its own — it only decides whether an already-computed finding is worth surfacing again. Wired into `js/app-shell/main.js`'s existing Watch Folder ingest path (`watchIngestAndValidate`) behind a new, disabled-by-default `semanticDriftWatchdog` flag (land dark); when on, a new drift finding renders as a small `validation-status` line (reusing the existing `pass`/`warn`/`fail` tones — no new CSS tone invented) beneath the file's status row, and the watchdog's memory is cleared whenever a new folder is connected. A native, OS-level (Tauri/Rust `notify` crate) file-watch trigger for the desktop shell was explicitly scoped out and tracked as a follow-up in `docs/tech-debt-tracker.md`, since this sandbox has no Rust toolchain to compile/verify one and Watch Folder itself is browser-only today; `drift-watchdog.js` was designed trigger-agnostic so that follow-up can reuse it unchanged. New `npm run test:driftwatchdog` (`test/drift-watchdog.test.mjs`, 32 tests covering summarization, fingerprint stability/order-independence, per-file de-duplication including pass→fail and fail→different-fail transitions, `.clear`/`.clearAll` re-arming, and malformed-input safety). Why: an ambient, always-on watcher that re-nags about a drift the user already saw (and hasn't changed) trains users to ignore it — surfacing only genuinely NEW drift signals keeps the alert meaningful.
- Add Metric Contracts Batch 3 (the confirm gate — the safety-critical piece): new `js/metrics/metric-contract-confirm-gate.js`. `proposeContractChange({metricId, currentMetric, candidate, proposedBy, reason})` builds a plain, inert PROPOSAL object — pure data construction, zero side effects, nothing written anywhere — which is the only thing an AI-agent caller in this codebase can produce with respect to a metric contract. `buildProposalDiffContent()` reuses Batch 2's exact `buildDiffViewContent` unmodified so a pending proposal renders pixel-for-pixel like a past human edit. `approve({proposal, contractRegistry, metricRegistry})` is the ONLY function in this whole codebase that can call `MetricContractHistory.recordVersion()` with `source: 'agent-proposed'` — it also applies the same change to the metric's live definition so the two never drift apart — and it only runs from the one Approve button `renderConfirmGate()`'s DOM presenter renders; there is no auto-approve timer, config flag, or bypass of any kind. `approve()` is idempotent (approving an already-applied proposal twice never double-appends history) and refuses cleanly without a registry, a metric registry, or on an already-rejected proposal. `reject({proposal, note})` writes nothing anywhere, ever, and is likewise idempotent and cannot retroactively undo an already-applied proposal. This directly reaffirms and tests DATAGLOW's hard autonomy-safety rule: an agent may propose, a human must approve every mutating action individually — the concrete cautionary precedent being the April 2026 incident where a Cursor AI agent deleted a company's entire production database and backups in 9 seconds with no confirmation prompt ([Euronews](https://www.euronews.com/next/2026/04/28/an-ai-agent-deleted-a-companys-entire-database-in-9-seconds-then-wrote-an-apology)). `renderConfirmGate()` shows the Batch 2 diff view plus two EQUAL-weight buttons (Approve/Reject, matching this project's established never-nudge-toward-accept pattern from the conversational pack builder); nothing in `js/app-shell/main.js` calls it yet — no AI agent in the running app can generate a real proposal through this gate today, this batch only builds and tests the gate itself. Tests: `npm run test:metriccontractconfirmgate` (16 passing, pure logic — propose/approve/reject correctness, idempotency, all four refusal paths, and an explicit end-to-end safety scenario proving repeated reads/renders of a pending proposal never mutate anything), added to the existing `metric-contracts` CI job alongside Batches 1 and 2.

- Add the Personal Data Bill of Materials: a one-click, fully offline "ingredient label" for a dataset, composing what DATAGLOW already knows about it into one signed-digest artifact rather than introducing any new inference or scoring. Why: give a user a single, portable, self-verifying record of exactly what a dataset is, where it came from, and whether on-device AI touched it — the kind of manifest a data engineer would want attached to any dataset before trusting or forwarding it — without adding any new trust claim beyond what DATAGLOW already computes and without ever acting on the user's data on its own.
