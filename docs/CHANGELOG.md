- Fix four independent, previously-confirmed bugs in one pass, each guarded by a fail-first regression test (verified to fail against pre-fix code and pass after) run by the new `batched-bugfixes` CI job. **Bug 1 (provenance):** DuckDB-WASM's `registerFileBuffer` transfers/detaches the `ArrayBuffer` it is handed, and the file-load path hashes those same raw bytes for the Chain of Custody — so if the engine detached the caller's buffer the later hash threw and the audit trail silently recorded nothing. `js/app-shell/duckdb-engine.js` now hands the engine an independent copy via a new exported `duckdbBytes()` (`.slice()`), decoupling the engine's transferable buffer from the caller's bytes so hashing (and any retry/re-read) stays valid regardless of call order. **Bug 2 (Benford's Law):** the CORE eligibility gate in `js/validation/validation.js` now recognises pure 0/1 flag columns itself (skip cause `binary_flag`) rather than relying on a domain pack's binary rule to relabel them — previously, with `pack:'none'`, a boolean flag fell through to the generic `narrow_range`/`small_sample` skip. **Bug 3 (Narrative Consistency):** a proportion column (every value in [0,1]) narrated as a PERCENTAGE (mean 0.30 → "30.0%") is now accepted instead of flagged as a false mismatch, by adding the ×100 percentage forms of the mean/min/max to the accepted-number set. **Bug 4 (SQL editor):** the plain textarea gains lightweight, dependency-free, zero-network syntax highlighting — a new pure, Node-testable tokenizer (`js/app-shell/sql-highlight.js`) paints a `<pre>` overlay behind a transparent-text textarea (`tok-keyword`/`tok-string`/`tok-comment`/`tok-number`/`tok-function` spans, HTML-escaped) — plus structured DuckDB error formatting (kind / detail / actionable hint) replacing the raw error dump. CodeMirror/Monaco were rejected: both would mean a CDN runtime dependency (forbidden for the zero-upload core app) or vendoring a bundle that needs a build step this project intentionally omits. Tests: `npm run test:bugfixes-layers` (native DuckDB, Bugs 2–3) and `npm run test:bugfixes-editor` (plain Node, Bugs 1 & 4). Why: guarantee the provenance audit trail is never silently empty, stop two false/again-crash validation findings, and make the SQL tab readable without breaking the local-first, no-build constraints.
- Add the Gen 40 Domain-Pack Plugin Architecture: convert the domain packs (`none`, `healthcare`, `retail`, `finance`, plus the OMOP and FHIR healthcare profiles) from code pasted into the single shared `js/validation/domain-physics.js` file into self-contained, manifest-declared plugins under a new `js/packs/` tree, so adding/updating/removing a pack is a new file plus one registration line and two packs never collide on the same file again. This is a pure architectural refactor — behaviour is assertion-for-assertion identical (the plugin modules reference the *same* runtime pack objects, proven by a legacy-vs-plugin identical-output test per extension point). New `js/packs/extension-points.js` names the closed vocabulary of six stable extension points (validation-rules, cleaning-fixes, anomaly-detectors, teaching-notes, sample-datasets, vocabulary); a pack manifest declares which points it fills and the loader rejects any unknown point. New `js/packs/pack-network-guard.js` makes the zero-upload guarantee enforced rather than conventional: a static source scan (`scanSourceForNetwork`/`assertNoNetwork`) rejects any pack whose code references a network primitive (`fetch`, `XMLHttpRequest`, `WebSocket`, …) and a runtime trap (`runWithNetworkDenied`) shadows those globals as defence in depth — the new `pack-architecture` test includes a test-only rogue pack that tries to `fetch()` and passes only because the guard catches it, and it scans every shipped pack file. New `js/packs/pack-registry.js` (`loadBuiltInPacks`) validates each manifest (id↔pack.name, semver version, industry, capabilities ⊆ extension points, and rejects any inter-pack dependency so packs stay isolated) and assembles the `{id → pack}` map the engine installs. `js/validation/domain-physics.js` gains an additive swappable pack-source seam (`setPackSource`/`getPackByName`/`registerRuntimePack`); the migration lands behind a `pluginPacks` flag in the existing `flags.manifest.json` (read by `js/build/build-flags.js`) wired into `js/app-shell/main.js`'s capability bootstrap, so it can be rolled back by flipping one flag. The Validate tab now surfaces a "Loaded domain packs" audit disclosure (id / version / extension points / sample-data license & provenance), mirrored in `TRUST.md`. Registered as the `domain-pack-plugins` capability in `capability-map.manifest.json` (with `docs/capability-map.md`), run in CI by the new `pack-architecture` job (`npm run test:packs`). Why: make each domain pack an independent, auditable, offline-guaranteed unit so the pack surface can grow without packs stepping on each other or the core engine.
- Complete the Context Engine (Gen 34 Stage C + D) with two additive, dependency-free features. **Stage C — Teach As You Clean** (`js/teaching/micro-lessons.js`): an optional one-line "why this matters" micro-explanation attached to every validation finding — all 20 layers plus the Red Team self-test, each domain-pack reinterpretation (healthcare/retail/finance), and the finer Unit Test (negative/future-date/blank-key/duplicate/null-ref) and Benford skip-cause sub-findings. Limitation (also noted in the PR): the hand-written healthcare pack uses bespoke predicates (`isSensitiveCategory`, wholesale-date-shift detection) that aren't factory-expressible, so export is scoped to the descriptor-based packs (retail, finance, and any imported pack); the healthcare and empty `none` packs are non-portable and report a clear reason. Why: teach users the reasoning behind each check as they clean, and let a team share its domain reinterpretations as a plain file without any hosted infrastructure or new trust surface.
