Multivariate Outliers

          

            Approximate on-device methods. Mahalanobis distance (Mahalanobis 1936) and Isolation Forest
            (Liu, Ting & Zhou 2008) flag rows that are unusual across several numeric columns at once.
            These are heuristics, not ground truth.
          

          

        

      


      

        

          

            Predictive Anomaly Scoring
            ML-based holistic score · not a validation layer
          

          

            An on-device, unsupervised k-nearest-neighbours outlier score (kNN distance, Ramaswamy et al. 2000)
            over Gower distance (Gower 1971), so it works across numeric and categorical columns at once.
            It learns the normal shape of the currently-loaded dataset and flags whole rows whose
            combination of values is unusual — holistic anomalies the single-column rules and the numeric-only
            Multivariate Outliers panel can miss. This is a statistical outlier detector personalised to
            this dataset's own distribution, not a general AI — and unusual is not the same as wrong: a flag may
            be a legitimate rare case, not an error. It is a complementary capability, separate from the 20 validation layers.
          

          

        

      


      

        

          
SPC Control Charts & Process Capability

          

            Shewhart 3σ control limits (Shewhart 1931) and the Cpk capability index (Six Sigma).
            Points outside the upper/lower control limits (UCL/LCL) signal a process out of statistical control.

Data Provenance Trail (Chain of Custody)

            

              
              
              
              
              
              
              
            

          

          

            A tamper-evident, SHA-256 hash-chained record of every transformation from raw file load onward.
            Each step's hash includes its parent's hash, so any later edit or reordering breaks verification — an audit trail for HIPAA provenance.
          

          

          


          

            
Cell-level Data Blame

            

              "Who/what changed this — and why?" Pick a column to see every recorded transform that touched it, in order, replayed from the append-only provenance chain above. Fully offline; nothing leaves your browser.
            

            

              
            

            

          

        

      


      

        

          

            
De-identification Verifier (HIPAA Safe Harbor)

            

              
              
            

          

          

            One click runs the 18 HIPAA Safe Harbor identifier categories against your columns and sampled values, scores a basic re-identification risk from the indirect identifiers present, and produces a SHA-256-signed attestation. A screening aid, not a certification — it can miss identifiers and can over-flag. Runs entirely in-browser against DuckDB-WASM; no upload.

Distribution Fingerprint Drift

    

      When enabled, DATAGLOW saves a tiny statistical summary of each column
      (mean, standard deviation, min/max, null rate, and distinct-value ratio) on this device only,
      so that a future file with the same schema — e.g. next month's export — can be checked for
      distribution drift. Only these summary numbers are stored — never your raw rows,
      and nothing is ever uploaded. Off by default.
    

    
    

    
    

      Trend-aware alerts. With persistence enabled, DATAGLOW also keeps a short,
      time-ordered history of each column's mean, missingness rate, and top-category share
      across uploads and projects the expected next value forward using
      Holt's linear exponential smoothing (Holt 1957). A new upload is flagged when it falls
      outside a confidence band sized by the method's own forecast error — catching drift away from the
      expected trajectory, not just away from a static baseline. This needs at least a few prior uploads
      of the same schema before it activates; until then the static drift check above applies.
    

    

    

    
Self-Learning Validation Rules

    

      DATAGLOW can learn from your own corrections — every time you apply or reject a
      suggested merge, or dismiss a validation flag — and use that to rank the flags you're most
      likely to care about first. It records a small labeled example (which check fired, the
      column type, and whether you acted on it or dismissed it) — never your raw cell values.
    

    

      What this is: a simple on-device statistical learner (a textbook
      logistic-regression model) that personalizes to your patterns. It is
      not a general-purpose AI, neural network, or deep-learning system, and it starts
      knowing nothing until you've made several corrections (at least 10). It only ever
      ranks and highlights — it never auto-edits your data.
    

    
    
    

    

    
Adaptive Layer Prioritization

    

      DATAGLOW can also learn which of the 20 validation layers tend to catch real issues for
      your data, and reorder the Validate tab so those layers surface first. Layers
      whose flags you keep acting on rise; layers you keep dismissing as noise sink.
    

    

      This only changes the order and adds a badge — it never hides, disables, or changes what any
      layer checks. Every one of the 20 layers still runs and stays visible. The score is a simple, explainable
      Beta-Binomial confidence (Laplace's rule of succession) with recency weighting — not a
      black-box model — so a layer with no feedback stays exactly where it is (it needs at least
      5 accept/dismiss actions before anything moves).
    

    
    
    

    

    
Federated Fingerprint Learning (research preview · opt-in)

    

      Optionally join other opted-in DATAGLOW users to collectively improve the shared
      fingerprint / pattern detection. Your browser trains a tiny model on your own
      validation feedback and shares only privacy-protected weight updates with peers —
      never your raw data, and never even your individual update in the clear.
    

    

      How your privacy is protected (all on top of a peer-to-peer WebRTC exchange):
    

    
    
      
  • Secure aggregation (pairwise masking, Bonawitz et al. 2017) — peers' masks cancel in the sum, so only the group average is ever readable.
    • 
      
  • Differential privacy (DP-SGD style, Dwork et al.) — each update is clipped and gets calibrated Gaussian noise before it leaves your device.
    • 
      
  • Minimum cohort — an update is only aggregated when at least 3 peers contribute to a round.
    • 
    

    

      Peer discovery uses a short-lived, self-expiring GitHub coordination file (WebRTC signaling only). If no peers
      are reachable it drops a masked, noised update there for a future peer — so it still helps with only 1–2 users.
      GitHub never sees raw data or any readable individual update. If WebRTC is unavailable or you're
      offline, DATAGLOW simply behaves as if this feature were off.

Red Team Mode

      
    

    

      Loads a built-in, intentionally broken golden dataset and runs all 20 validation layers against it. Every layer must catch its target issue — that's how you know the validation system is real, not decorative.
    

    
    

      v2: synthesize a fresh adversarial dataset that matches your currently-loaded dataset's schema — same columns and types, seeded with near-duplicate spellings, cross-column logic violations, duplicates, nulls, outliers, future dates, and schema-mismatched values — then report which seeded categories the layers catch.
    

    
    


    

    
Synthetic Adversarial Twin

    

      Generate a synthetic version of your currently-loaded dataset that preserves each column's overall shape but contains no real rows. Differentially-private (Laplace) noise is added to every histogram / category count before resampling, so the synthetic data is safe to share for demos and testing.
